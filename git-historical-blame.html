                • pretty big idea from this (and i've had a thought or 2 about this quite recently, pretty sure it was related to [[git-stacked-rebase]] & the diff / pull request review thoughts, this just sparked it further) -- what if we had [[a way to explore code in 3D]]?
                  • what i imagine is that you'd go thru the file's history, compute how much it has expanded, and then start from the bottom of the stack / layer / whatever name fits best.
                    • on the first stack, you'd put the file's contents of the very first commit.
                    • the contents would probably be quite far away from each other.
                      • this is because in further layers, we'd want to display how the contents changed w/ further commits, and if e.g. some block/paragraph got extended, to visualise it by not the whole file growing, but that part of the content growing, and so the previous layers would have to be already prepared & spread out enough to accomodate for the new changes __for that block/paragraph__, without affecting the position of other blocks/paragraphs.
                        • this is just to make the diff clearer / exploration more meaningful -- you're not only able to explore how the whole file changed, but rather - how some small, individual part of the file changed
                          • https://firebasestorage.googleapis.com/v0/b/firescript-577a2.appspot.com/o/imgs%2Fapp%2Fkipras-g1%2Fcl0Th6BIb1.psd?alt=media
                          • ![](https://firebasestorage.googleapis.com/v0/b/firescript-577a2.appspot.com/o/imgs%2Fapp%2Fkipras-g1%2F7nbv1Lgyin.png?alt=media&token=7b9629e8-cd46-4884-bd96-b5cf4558a889)
                            • i'm not good w/ drawing, but here you can see change over time. and this is only additions. here, the brightness changed, but ideally what i have in mind is for the depth to change - for it to be like actual layers / stacks / whatever, like taller & shorted buildings / skyscrapers, and you can jump down deeper to explore the depths of where the current latest version came from
                            • the hardest part is probably going thru all the commits & figuring out which parts blocks (lines close by) relate to each other, so that you can split them up further apart in earlier versions -- it's hard to know where to split.
                              • probably a single function is a good boundary, but ofc. not everyone writes just pure functions.
                                • tho even if we could perfectly split functions, would it be the most representative way of what things are related to each other?
                                  • i think that while trying to develop this, you might discover the best way to write programs, given that the measurement of how good the program is is how easy is it to understand what relates to what, which, if it makes it faster/easier to understand the program itself, would be a very good thing to strive for / to encourage in general.
